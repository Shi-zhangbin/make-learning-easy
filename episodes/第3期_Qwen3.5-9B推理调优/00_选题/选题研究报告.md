### 核心参数对照表（调优对比用）

| 参数 | 调优前 | 调优后 | 影响 |
|------|--------|--------|------|
| --tensor-parallel-size | 4 | 4（固定） | 使用4张NPU并行推理 |
| --max-model-len | 5000 | 2000 | 缩短上下文→降低显存占用 |
| --max-num-seqs | 128 | 64 | 减少并发→降低OOM风险 |
| 首token延迟 | ~800ms（基准） | ~400ms（优化后） | 响应速度提升 |
| 吞吐量（tokens/s） | ~120 | ~180 | 每秒输出增加 |

## 三、核心类比体系

### 类比1：汽车改装 → 推理调优（全新设计）

本期的王牌类比，贯穿概念+实操全篇。

| 技术概念 | 类比对应 | 说明 |
|---------|---------|------|
| 模型（Qwen3.5-9B） | 一辆高性能跑车 | 出厂就很快，但需要调教才能发挥真正性能 |
| vLLM推理框架 | 赛车的发动机管理系统（ECU） | 控制引擎怎么喷油（怎么推理）才能快又稳 |
| tensor-parallel-size | 引擎缸数（4缸→4张NPU并行） | 更多缸并行工作，但更吃油（显存） |
| max-model-len（上下文长度） | 油箱容量 | 油箱越大，能连续跑的路越长（处理的文字越多） |
| max-num-seqs（并发数） | 同时载客数 | 一辆车能同时坐几个人，越多越挤（越可能OOM） |
| 首token延迟 | 0-100加速时间 | 踩下油门到推背感传来的时间 |
| 吞吐量（tokens/s） | 极速（最高时速） | 跑起来后每秒能跑多远 |
| 显存OOM | 油箱爆了 | 装太多开车直接趴窝 |

**话术示例：**
> "一台车出厂已经很快了——但它用的是通用调校。如果你是赛车手（运维开发者），你会自己调ECU（vLLM参数），让这台车在赛道上（你的硬件上）发挥极致。今天的实操，就是带你们当一次汽车的调校师。"

### 类比2：体检 → 性能指标（复用优化）

| 技术概念 | 类比对应 |
|---------|---------|
| 首token延迟 | 血压测量：一按下去多久出第一个数字 |
| 吞吐量 | 肺活量测试：一口气能吹多远 |
| 显存使用 | 体重秤：看看占了多少资源 |
| 调优 | 调整饮食+运动方案，看体检指标变化 |

**话术示例：**
> "跑模型之前，我们先给模型做个'体检'——测它启动多久才回第一句话（首token延迟）、每秒能吐多少字（吞吐量）、吃掉多少显存空间。调完参数再测一次，你就知道你的'调校方案'有没有用。"

### 类比3：厨房 → 环境准备（复用第2期）

| 技术概念 | 第2期类比 | 本期动作 |
|---------|-----------|---------|
| docker pull | 从宜家下单 | 拉镜像=下载"厨房套装" |
| docker run + --device | 在阳台搭外卖间+插USB-C | 创建容器=布置独立厨房 |
| 容器内启动vLLM | 厨房里开火做饭 | 模型推理=开始做菜 |

**话术示例：**
> "第2期我们认识了厨房（容器）和厨具（NPU），今天我们要真正'开火做饭'了。打开煤气灶，食材下锅，看看能不能炒出一盘好菜。"

## 六、模型部署参考命令

本期实操全程基于以下命令设计：

```bash
# ① 检查NPU
npu-smi info

# ② 拉取镜像并启动容器
docker run -it --rm \
  --device /dev/davinci0 \
  --device /dev/davinci1 \
  --device /dev/davinci2 \
  --device /dev/davinci3 \
  -p 8010:8010 \
  -v /root/.cache/modelscope:/root/.cache/modelscope \
  vllm-ascend-local:latest \
  bash

# ③ 容器内启动vLLM推理服务（调优前）
vllm serve /root/.cache/modelscope/hub/models/vllm-ascend/Qwen3.5-9B/ \
  --served-model-name "qwen3.5" \
  --host 0.0.0.0 \
  --port 8010 \
  --tensor-parallel-size 4 \
  --max-model-len 5000 \
  --max-num-seqs 128 \
  --trust-remote-code \
  --enforce-eager

# ④ 测试推理
curl http://localhost:8010/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "qwen3.5",
    "messages": [{"role": "user", "content": "你好，请用50字介绍你自己"}],
    "max_tokens": 100
  }'

# ⑤ 调优后重启服务（缩短max-model-len，调整并发）
vllm serve /root/.cache/modelscope/hub/models/vllm-ascend/Qwen3.5-9B/ \
  --served-model-name "qwen3.5" \
  --host 0.0.0.0 \
  --port 8010 \
  --tensor-parallel-size 4 \
  --max-model-len 2000 \
  --max-num-seqs 64 \
  --trust-remote-code \
  --enforce-eager
```

## 七、风险清单

| 风险级别 | 风险 | 描述 | 缓解措施 |
|---------|------|------|---------|
| 🔴 **高** | 模型加载时间过长 | Qwen3.5-9B参数大，加载入显存需数十秒，影响观看体验 | 录屏时加速等待过程+字幕说明"正常情况要等X秒" |
| 🔴 **高** | vLLM Ascend兼容性问题 | vllm-ascend可能不兼容最新版Qwen3.5 | 提前在真实环境验证，录制备选方案 |
| 🟡 中 | 命令太长看不清 | vllm serve命令参数多，观众容易眼晕 | 分屏：左命令（放大字体）右效果；每个参数单独一行标注 |
| 🟡 中 | 调优效果不够明显 | 参数微调后性能变化不大，观众觉得"没用" | 选择差异大的参数组合（max-model-len从5000砍到2000） |
| 🟡 中 | 纯小白在第2期和第3期之间断了 | 只看第2期的观众隔太久忘了概念 | 开场30秒快节奏回顾三个类比，前置全流程地图 |
| 🟡 中 | 实操节奏太快，观众跟不上 | 命令太多、代码太长 | 每步操作后停顿+截图，"你现在看到的输出应该长这样" |
| 🟢 低 | docker pull镜像太大 | 镜像下载慢，影响演示流程 | 提前拉好，录屏只用"已完成拉取"的画面 |
| 🟢 低 | 专业名词堆砌 | vLLM/tensor-parallel/吞吐量等术语混用 | 每个名词首次出现立刻配类比解释 |
| 🟢 低 | 第3期内容与第2期重复 | 观众觉得"又讲一遍Docker，浪费时间" | 明确说"今天讲第2期的升级版——上次看懂图纸，这次真开干" |
| 🟢 低 | 时间超限 | 实操环节不可控，容易超过20分钟 | 实操按优先级排序：核心4步必须完成，调优对比可快剪 |
