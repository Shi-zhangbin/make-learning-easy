💡 API 的本质：一个「中间人」

API 就是两个程序之间约定好的一套沟通规则。你不需要知道对面怎么实现的，照着规则发请求就行了。

☎️
打电话的比喻
你不需要知道电话局怎么转接的，你只需要知道对方的号码，拨号，等接通就行了。API 就是这个「电话号码」+「通话协议」。
🎮
游戏手柄类比
你按 A 键跳跃，按 B 键攻击。你不关心手柄内部电路怎么处理的，你只关心按了什么键、得到什么结果。API 就是游戏的「按键映射表」。
👨‍💼
餐厅服务员
你跟服务员点菜，服务员告诉后厨，后厨做完菜服务员端给你。你没有直接跟后厨打交道——API 就是那个服务员。
🔄
程序 A → API → 程序 B

🎯 REST API：互联网的通用语

🌐
GET /users → 返回用户列表
POST /users → 创建新用户
PUT /users/1 → 更新用户
DELETE /users/1 → 删除用户
RESTful URL 设计示例
📮
GET — 查
向服务器请求资源，就像翻一本字典。幂等，安全。
只读
✉️
POST — 增
创建新资源，每次调用都产生一个新东西。发一次和多发一次不一样。
非幂等
🔄
PUT — 改
替换整个资源。你要把一页纸全部擦掉重写。
幂等
🗑️
DELETE — 删
删除资源。删了就是删了，再删一次也是同样的结果。
幂等

REST vs GraphQL：谁更香？

同样的需求，两种不同的 API 设计哲学。没有银弹，但各有主场。

REST — 资源驱动

  • 每个 URL 代表一个资源，语义清晰
  • 缓存友好：HTTP 缓存直接用
  • 工具链成熟：Swagger, Postman 全家桶
  • 过度获取（Over-fetching）：一个接口返回整张表
  • 版本管理麻烦：v1, v2, v3 满天飞
  • 多个资源需要多次请求，N+1 问题

    • GraphQL — 查询驱动

  • 客户端精确指定需要的字段，不多不少
  • 一次查询拿多个资源，告别 N+1
  • 类型系统：Schema 即文档，自省能力强
  • 缓存复杂：不像 HTTP URL 那么好缓存
  • 查询复杂度难控制：一个深层嵌套查垮服务器
  • 文件上传等二进制操作支持较弱
    • 选 REST：团队小、CRUD 为主、需要强缓存
    选 GraphQL：前端场景复杂、多端共用、对数据粒度敏感

    ⚠️ API 设计的 4 个常见翻车现场

    这些都是我在 code review 里真实见过的。每一个都有血有泪。

    🚮
    返回 200 但报错
    code 200, body 里写 {"error": "token过期"}。前端每个请求都要先解析 body 看有没有错误。HTTP 状态码不是摆设！
    📏
    接口膨胀
    /getUser、/getUserInfo、/getUserDetail 三个接口做的事一模一样。不同开发者各写各的，没人肯删。一条规范能省 3 个人天。
    🔓
    权限裸奔
    只做了前端按钮隐藏，后端的 API 没有任何鉴权。用 Postman 直接调 /admin/deleteAll 就能删库。这种事发生得比你想象的频繁。
    🐌
    N+1 查询地狱
    查询订单列表 → 每个订单再查一次用户信息。如果 100 个订单就是 101 次数据库查询。加个 JOIN 或者用 GraphQL 就能解决。

    🔧 修复方案：桥接 PageSpec 到 T6

    如上面 10 个布局所示，pagespec.py 已经具备完整的页面结构推导能力（布局自动选择、内容解析、层数保障）。但 t6_compositions.py 完全没有使用它。需要做三件事：

    1
    T3 存每页叙述文本到 timeline.json
    在 v3/steps/tts.py 中，把解析出的每页口播正文存入 timeline.json 的 slide 对象（key: narration）。这样 T6 就能拿到每页的文字内容。
    2
    T6 调用 pagespec.build_all_pages() 生成 PageSpecs
    在 CompositionHandler::execute() 中，读取 timeline.json（此时已有 narration）后调用 pagespec.build_all_pages(slides, images) 获得结构化的 PageSpec[]。这一步自动完成布局选择和内容分段。
    3
    写转换器：PageSpec → Elements 数组
    把每个 PageSpec 转换成 _render() 能吃的元素列表（badge / heading / card-row / grid-2x2 / code / split / quote / image 等）。这个转换器约 80-120 行，覆盖全部 10 种布局的分支映射。
    💡 改动面
    修改 2 个文件：v3/steps/tts.py（~10 行追加 narration 字段）、v3/steps/t6_compositions.py（~100 行新代码：提取 narration + 调 build_all_pages + PageSpec→elements 转换器）。
    新增 0 个文件。pagespec.py 和设计预设 YAML 完全复用。不动现有 gate 逻辑。
